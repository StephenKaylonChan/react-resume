### 4. AI 和自动化相关（重点）

#### 语义相似性分析系统 —— 三代 AI 分析引擎的渐进式演化

**业务问题**：未入库清单中存在大量同义表述（"混合砂浆" vs "预拌砂浆"），手工识别成本高，重复创建浪费清单资产。

**第一代（V2 - Embedding 聚类）**：

1. **关键词预分组**（similarity_grouping.py）
   - 从 13,548 条未入库清单中提取关键词
   - 使用 Embedding 向量相似度（阈值 0.82）快速聚类，减少 LLM 调用
   - 输出：约 50-100 个粗粒度分组

2. **LLM 精细分析**（similarity_service.py）
   - 对每个分组调用 LLM 判断语义相似性
   - 并发控制：最多 10 个并发请求（可配）+ 0.1s 最小间隔
   - 输出：JSON 格式的同义词分组 `{main_name: [synonyms]}`
   - 持久化：保存到 `similarity_result.json`，支持增量更新

3. **线程安全和进度跟踪**
   - 使用 threading.Lock 保护运行时状态
   - 进度持久化到 `similarity_status.json`，支持任务中断恢复
   - 前端可实时查询分析进度

**第二代（V3 - 层级化维度分解）**：

1. **纯净名称提取**（similarity_v3.py）
   - 去除变体信息（颜色、材质、规格、品牌、位置等）
   - 保留核心商品类型名称
   - 示例：`黑色不锈钢混合砂浆` -> 纯净名 `混合砂浆` + 变体 `{color: 黑色, material: 不锈钢}`

2. **变体维度识别**（12 个维度配置）
   - 颜色：16 种 | 材质：32 种 | 品牌：50+ 种
   - 位置：卫生间、厨房、大堂、室外等
   - 商品类型：补差、主材、增补、漏项、费用等
   - 等级/规格：A/B/C 级、DN/Phi/尺寸等

3. **同义词合并**
   - 先按纯净名称分组
   - 再用 Embedding（0.90 阈值）+ LLM 判断同义词
   - 最后按变体维度聚合展示

**第三代（V4 - LLM 结构化提取）**：

1. 用 qwen3-max（更强的指令理解能力）替代 qwen-plus
2. 单条 Prompt 一次性提取多个维度（纯净名、颜色、材质、品牌、规格等）
3. 批量异步处理：并发 10、每批 100 条，预计耗时 30-50 分钟
4. 结果持久化到 MySQL，支持增量更新

#### Prompt 工程细节

**V2 分组判断 Prompt**：
```
分析这些清单名称是否属于同一类商品（语义相似）:
[清单列表]
返回 JSON: {"is_similar": true/false, "reason": "..."}
```

**V3 纯净名称提取 Prompt**：
```
商品名称: "黑色不锈钢混合砂浆"
提取:
1. 纯净名称（核心商品名）
2. 颜色
3. 材质
4. 规格
...
返回 JSON 结构
```

#### 错误处理与容错

- JSON 解析失败：用平衡括号计数算法（比贪心正则安全）从响应中提取 JSON
- API 限流（429）：指数退避重试（2s、4s、8s）
- Embedding 失败：降级处理，返回随机小向量（避免 NaN）
- 网络超时：120s 超时设置，可配置

#### 60 个预计算任务的持久化缓存（cache_persistence.py）

- 启动时尝试从 MySQL t_analysis_cache 表加载结果（<0.5s）
- 如无缓存，逐个计算并写入 DB
- reload-data 后重新计算全部 60 个任务
- 任务覆盖：四维度分析 x5、问题追溯 x1、总览 x1、未入库/重复 x2、Drill-through x19、Detail 穿透 x30

### 6. 产品设计决策

**技术选型理由：**

- FastAPI（非 Django/Flask）：内置异步支持、自动 OpenAPI 文档、Pydantic 类型验证
- MySQL（非 PostgreSQL）：数据规模 13,548 行不需要 PostgreSQL 高级特性，部署更简单
- 阿里云百炼（非 OpenAI）：国内合规性、成本更低、OpenAI API 兼容（可迁移）
- 单 Gunicorn Worker：应用状态在内存中，分布式会话管理复杂度高

**数据流设计：**

```
初始化：Docker 启动 -> warmup_cache() -> pandas 加载 Excel(~11s)
  -> 维度预处理(~3s) -> DB 缓存注入(<0.5s) -> 服务就绪

查询流程：用户请求 -> FastAPI 路由 -> @cached 装饰器检查内存缓存
  -> Hit: 直接返回 <1ms
  -> Miss: 计算 -> 注入缓存(TTL=365天) -> 返回 JSON

语义分析：POST /start -> 后台任务 -> 提取纯净名称 -> Embedding 聚类
  -> LLM 精细判断 -> 持久化结果 -> 前端轮询 GET /status
```

**产品思维体现：**

- LLM 限流自动重试，用户无感知
- Embedding 失败降级，分析不中断
- 缓存键自动排序（build_cache_key），参数顺序无关
- 数据版本管理，reload-data 后自动重算全部缓存
- LLM 服务抽象（call_llm / call_llm_json），换模型不改业务代码

## 项目 3：招标信息自动化抓取

### 1. 这个项目是干什么的（一句话）

为战略市场部构建商机早期发现平台，通过多源爬虫自动采集招投标公告与客户扩张动向，并用 LLM 自动摘要和事件分类，帮助市场人员比竞争对手更早发现潜在客户信号。

### 2. 我负责的具体模块（全栈独立设计 + 主导开发）

1. **爬虫采集框架**
   - 公众号采集：纯 HTTP 扫码登录 + appmsg API 分页采集 + WechatSession 状态机
   - 网页爬虫框架：BaseCrawler ABC + GenericHtmlCrawler CSS 选择器驱动 + 适配器注册表（已支持 4 个网站）
   - 客户动向采集：撒网式 aliases 匹配 + 事件类型自动分类 + 3 源配置（36 氪/红餐网/联商网）

2. **LLM 集成服务**（OpenAI-compatible）
   - 设计为无依赖 + 环境变量配置，兼容 OpenAI/通义/Deepseek/Ollama
   - 批处理管道：pending 状态追踪 + 防重复处理 + 错误重试
   - Prompt 工程：系统提示词 + 用户提示词双层设计，输出 2-3 句中文摘要

3. **定时调度系统**（APScheduler）
   - 6 个活跃任务：公众号采集 + 网站采集 + 客户动向采集 + LLM 摘要 + 每日采集日报 + 每日跟进日报
   - Cron 触发器 + 随机抖动（防突发）+ 并发锁保护

4. **后端 API 层**（87 个端点）
   - 认证与授权：JWT + admin/user 两级角色 + 5 次/分钟登录速率限制
   - 招投标模块：14 个端点（列表/搜索/批量操作/去重）
   - 客户动向模块：18 个端点（多维筛选/事件分类/跟进管理）
   - 管理后台：42 个端点（源管理/采集触发/LLM 控制/用户管理）

5. **前端 UI 层**

   - React 19 + Vite + Ant Design 6
   - 品牌视觉：金色主题（#FFC000）+ 自定义深灰侧边栏

6. **数据模型与持久化**（16 张表）
   - 核心表：bidding_articles + customer_articles + monitored_companies
   - 状态设计：三态互斥（inbox/tracked/archived）+ AI 状态机（pending/processing/done/failed）
   - 索引优化：复合索引 + 单列索引

   - 数据库迁移：Alembic + 5 个生产迁移脚本
   - 生产安全检查：弱密钥拦截、CORS 限制、SQL 注入防御

### 4. AI 和自动化相关（重点）

#### LLM 摘要管道（核心创新）

- 位置：`backend/app/services/llm_service.py`
- 入口：`process_pending_articles(db)` -> 批处理 pending 状态文章
- 调用方式：
  - POST `/chat/completions`（OpenAI-compatible endpoint）
  - 系统提示：企业情报分析师身份
  - 用户提示：title + content（截断 3000 字）-> 2-3 句摘要（<=120 字）
  - 参数：model 可配、temperature=0.3（低随机性）、max_tokens=300
- 错误处理：单篇失败记录 ai_status=failed，继续处理下篇，不阻断整个 job
- 可配项：LLM_ENABLED / LLM_BASE_URL / LLM_API_KEY / LLM_MODEL 四个环境变量

#### 自动爬虫与事件分类

- 客户动向分类器：`classify_event_type()` 根据关键词集合自动分类
- 事件类型：融资（financing）/ 扩张（expansion）/ 管理层（management）/ 财报（financial_report）
- 监控公司的 aliases 实现灵活匹配：支持中文名 + 英文名 + 别名

#### 关键自动化工作流

```
定时触发 -> 公众号/网站/客户动向三源并行采集（CronTrigger + jitter）
  -> 采集后自动去重（UNIQUE URL）+ 入库
  -> 标记 ai_status=pending
  -> LLM job 定时扫描 pending -> 调用 API -> 存储摘要
  -> 前端展示 ai_summary + event_type，支持按事件筛选
```

#### 与 Agent/工具调用最相关的部分

- 爬虫适配器框架体现工具化思想：统一 interface（fetch_list / fetch_detail / parse）+ 多个实现（GenericHtmlCrawler / JsonApiCrawler / RssCrawler）
- LLM 服务的无 SDK 设计允许轻松扩展到任何 OpenAI-compatible 服务
- 定时任务与 API 触发分离，支持调度器驱动或 API 手动触发

### 6. 产品设计决策

**技术选型理由：**

- FastAPI：异步原生，支持 APScheduler asyncio 驱动

- LLM 无 SDK 设计：环境变量切换模型（OpenAI/通义/Deepseek/Ollama），解耦彻底

**数据流设计：**

```
采集源（公众号/网站/RSS）
  -> httpx 异步请求 + BS4 解析

  -> 去重（UNIQUE URL）
  -> bidding_articles / customer_articles

  -> ai_status = pending -> LLM job -> ai_summary + ai_status = done
  -> 前端展示 + 按事件筛选 + 跟进管理
```

**产品思维：**

- 单篇 LLM 失败不阻断整个 job（错误隔离）
- 三态互斥（inbox/tracked/archived）清晰的数据生命周期
- 爬虫适配器注册表，新网站只需配置 JSON
- 启动时检查 SECRET_KEY，防弱密钥上线
- APScheduler 并发保护 + 卡死自恢复

**数据流设计：**

```

  -> phase='cover': 显示 ModuleCover -> coverDuration 结束 -> setPhase('content')
  -> phase='content': 显示对应 Viewer
     -> Viewer 内部管理子阶段(stage)
     -> contentDuration 或 onComplete 回调结束
     -> nextIndex = (currentModuleIndex + 1) % 7
     -> 循环
```

**产品思维：**

- Ken Burns 效果（1->1.05 缓慢放大）模仿电影质感
