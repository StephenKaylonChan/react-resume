## 分析模板

新项目探索或已有项目更新时，按以下模板输出 7 个字段，追加到 `项目代码深度分析_5个项目.md` 中对应项目的章节。

```markdown
### [项目名称]

**1. 这个项目是干什么的（一句话）**
用产品视角描述：解决了谁的什么问题

**2. 我负责的具体模块**
列出所有功能模块，标注哪些是独立设计、哪些是主导开发的

**3. 技术栈（越具体越好）**
- 前端：框架、UI库、状态管理、路由等
- 后端：语言、框架、数据库、ORM等
- AI/LLM：用了哪些 API、哪些框架、哪些模型
- 部署/工具：Docker、CI/CD、云服务等

**4. AI 和自动化相关（重点）**
- 哪些功能用到了 AI/LLM/爬虫/自动化
- 具体是怎么实现的（调用哪个 API、用了什么 Prompt 策略、怎么处理输出）
- 和 Agent/工具调用/知识库最相关的部分是什么

**5. 可量化的数据（从代码和配置中推断）**
- 处理的数据规模（行数、文件数、接口数）
- 有没有批处理、分页、并发相关的代码
- 任何能体现规模或效果的数字

**6. 产品设计决策（从代码结构推断）**
- 为什么选这个技术栈而不是其他
- 数据流怎么设计的
- 有没有体现产品思维的设计（错误处理、用户体验、可扩展性）

**7. 简历可用的一句话项目描述**
格式：用 [技术栈] 开发了 [功能描述]，解决了 [具体问题]，实现了 [可量化成果]
```

## 探索方法论

Claude Code SubAgent 探索项目时，按以下优先级读取文件：

1. **README / 说明文档** — 快速理解项目定位
2. **根目录结构**（ls -la）— 了解项目组织方式
3. **依赖配置**（package.json / requirements.txt / pom.xml）— 确认技术栈和版本
4. **前端入口**（main.tsx / App.tsx / page.tsx）— 理解页面结构和路由
5. **后端入口**（main.py / app.py / Application.java）— 理解 API 架构
6. **核心业务组件和服务** — 深入模块逻辑
7. **AI 相关代码** — 搜索 `AI / LLM / Claude / OpenAI / agent / prompt / embedding`
8. **配置和部署文件**（Dockerfile / docker-compose / .env.example / vite.config）

### 探索深度级别

| 级别 | 场景 | 做什么 |
|------|------|--------|
| 快速扫描 | 新项目初次了解 | 读 README + 依赖 + 入口文件，输出 1-3-7 字段 |
| 标准探索 | 首次完整分析 | 完整 7 字段模板，含代码行数和 API 端点数 |
| 增量更新 | 项目有新功能 | 只读变更部分，更新受影响的字段 |

## 更新流程

### 场景 1：已有项目代码更新了

```
请帮我更新 [项目名] 的分析。
项目路径：[路径]
上次分析时间：[见 README 项目清单]

重点关注自上次以来的变更：
- 新增了哪些功能模块或 API 端点
- 技术栈有没有升级
- AI/LLM 相关功能有没有变化
- 可量化数据是否需要更新

更新 项目代码深度分析_5个项目.md 中对应章节，
并更新 README.md 项目清单中的「最后更新」日期。
```

### 场景 2：新增一个项目

```
请帮我分析一个新项目并补充进简历素材：

项目名：[名称]
路径：[路径]
技术栈：[简述]

按 docs/工作项目梳理/README.md 中的分析模板和探索方法论进行探索，
输出完整 7 字段分析，追加到 项目代码深度分析_5个项目.md 末尾，
更新 README.md 的项目清单和简历重点排序，
更新跨项目技术汇总。
```

### 场景 3：准备某个方向的简历版本

```
我要投 [岗位方向] 的简历。
请基于 docs/工作项目梳理/ 中的分析文档，
重新排序项目优先级，
为每个项目挑选最匹配该岗位的亮点，
生成一版定向的项目描述。
```

### 场景 4：定期全量刷新

建议每 1-2 个月做一次全量刷新，确保分析与代码同步：

```
请对 docs/工作项目梳理/README.md 中列出的所有项目做一次全量刷新。
用 SubAgents 并行探索每个项目，对比已有分析内容，
输出变更摘要（新增了什么、删掉了什么、数字是否变化），
更新 项目代码深度分析_5个项目.md 和 README.md。
```
