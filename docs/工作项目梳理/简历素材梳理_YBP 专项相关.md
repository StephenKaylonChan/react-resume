## 一、我实际负责了哪些开发工作

### 项目：YBP（YOMY BIM PLATFORM）—— 核心项目

**项目背景**：面向麦当劳餐厅装修项目的 BIM 工程量自动提取平台，打通 Revit 建模 → 标准清单 → 造价核算全链路，替代传统 CAD 人工计量方式。

**负责模块与具体任务：**

**① 业务流程梳理与文档主导**
- 完整梳理六阶段业务流程：族库上传 → 维护族库 → 配置清单关系 → 建模员下载使用 → 导出模型信息 → 系统匹配计算工程量
- 主导与开发合作方易达（Yida）的需求沟通，将业务逻辑转化为技术文档

**② 核心匹配逻辑设计（族 → 品目 → 清单链路）**
- 设计"清单视角匹配机制"：每条清单自带匹配条件（参数名 + 运算符 + 目标值），支持 AND 组合
- 设计区间匹配规则，解决参数值无穷变化问题（如风量 ≤ 2000 / 2000~4000 的分档清单匹配）
- 设计参数三级分类体系：基础参数（引用族参数）、衍生参数（区间匹配带出清单）、几何保护参数（只读，防止模型错误）

**③ 复合构件处理方案设计**
- 叠层墙：设计"BOQ 组"方案，通过 `GetStackedWallMemberIds()` 拆解为独立基本墙，每层分别匹配清单，复用基本墙配置，无需重复维护
- 幕墙：净面积 = 总面积 - Σ嵌板面积，明确插件与系统的计算分工
- 幕墙嵌板通过 `CurtainGrid.GetPanelIds()` 提取，嵌板可能属于墙/窗/门不同类别，设计类别专项匹配逻辑
- Assembly：明确为逻辑分组容器而非物理组合，厘清是否纳入 YBP 处理范围

**④ Demo 验收测试包设计**
- 选取 4 个典型麦当劳构件（水磨石地砖、水磨石踢脚、儿童软包底层、木纹防火板）设计 Demo 测试包
- 处理"衍生清单"复杂场景：儿童区基层和大堂基层来源不同（独立族 vs 叠层墙衍生），但因单价相同（134.23元/㎡），合并为一条清单项
- 确立验收基准总价：18,184.67 元，用于系统计算准确性对标

**⑤ 技术文档输出（多受众并行）**
- 为易达输出完整需求文档，同时服务三类读者：产品经理（完整需求）、开发（技术细节）、测试（快速数据对照）
- 输出内容包括：系统原型设计记录、数据速查表、Demo 说明文档、沟通邮件文本

**⑥ Revit API 学习与技术验证**
- 系统学习 Revit 2024.2 SDK，重点研究 Family / FamilySymbol / FamilyInstance 三层结构
- 使用 RevitLookup 插件实地验证元素参数，区分类型参数与实例参数
- 掌握叠层墙、幕墙、Assembly 的 API 访问模式（C# 代码层面理解）

## 三、遇到过哪些技术难题，怎么解决的

### 难题 1：叠层墙无法直接匹配清单

**问题描述**：Revit 中叠层墙是一个整体 Wall 实例，但业务上需要对每层独立计算工程量并匹配不同清单，如果在系统中维护一个"叠层墙品目"，会导致配置爆炸（每种组合都要单独配置）。

**解决方案**：
- 明确叠层墙是"容器"而非物理实体，通过 `GetStackedWallMemberIds()` 穿透容器，提取各子墙的独立 ElementId
- 系统层面**不维护叠层墙品目**，只维护基本墙品目
- 插件层自动拆解，每层基本墙独立走匹配流程
- 优势：建模员仍用叠层墙提效，系统自动复用基本墙配置

### 难题 2：参数值无穷变化无法精确匹配

**问题描述**：设备类构件（如新风机）的参数值（风量）是连续区间，不能为每个数值单独配清单。

**解决方案**：设计**区间匹配规则**，每条清单配置上下界（> 2000 AND ≤ 4000），支持多参数 AND 组合，彻底解决连续参数的匹配问题。

### 难题 3：同单价清单的合并问题

**问题描述**：Demo 包中，儿童区软包底层来自独立族实例，大堂踢脚底层来自叠层墙衍生计算，两者是不同来源，但单价相同（134.23元/㎡），业务上应合并为一条清单项。

**解决方案**：明确"衍生清单"的设计原则——**共享单价的清单可跨来源合并**，在清单层做 merge 而非在族/品目层处理，既保留计算逻辑清晰，又满足业务展示需求。

### 难题 4：几何参数被误改导致模型错误

**问题描述**：Revit 的几何参数（如墙体厚度、开口尺寸）影响模型实际形态，若系统运营人员在 YBP 后台修改这些参数，会导致插件回写 Revit 时破坏模型。

**解决方案**：设计**几何参数保护机制**，将几何类参数标记为"只读引用"，系统界面禁止编辑，只读展示，从架构层面防止此类错误。

### 难题 5：幕墙嵌板类别不统一

**问题描述**：幕墙嵌板通过 `CurtainGrid.GetPanelIds()` 获取，但嵌板可能属于墙、窗、门三种类别，不同类别对应不同清单匹配逻辑，不能用统一规则处理。

**解决方案**：确立"嵌板类别专项匹配"原则，先判断嵌板所属类别，再走对应的匹配分支。

## 四、工作方式和思维方式

### 需求理解方式：从业务出发，往技术落地

- 理解需求路径是：**麦当劳业务场景 → 建模员使用习惯 → 系统数据结构 → API 实现**，不是反过来从技术推业务
- 设计叠层墙方案时，优先考虑"建模员不想改工作方式"，再逆推系统如何适配

### 技术决策时考虑的因素

- **可用性优先于性能**（上级明确指导，当前数据量下 SQL 简单比查询快更重要）
- **系统可配置化**而非硬编码（参考 YDC 系统设计哲学，不写死逻辑）
- **多受众兼顾**：文档设计时同时考虑产品经理、开发、测试三类读者

### 主动提出的方案和改进

- 主动发现"参数视角匹配"的局限性（无法表达多对多关系），推动转向"清单视角匹配"
- 主动设计 Demo 验收包（而非等待开发方设计），并预算出精确的验收基准总价
- 主动将数十次对话内容归档为项目背景文档，解决跨项目知识传递问题

### 协作模式

- 对外（易达）：主导业务逻辑解释，要求技术实现方提供 ER 图并对照 YDC 参考架构
- 对内（昱姐）：带着具体设计方案请指导，而非带着问题请答案

## 五、入职至今能力变化最明显的地方

### 1. 技术理解深度：从"不懂就问"到"带方案来讨论"

**早期（2025年10月前）**：对 Revit 族/系统族概念混淆，对数据库设计几乎没有概念，学习 WPS 宏时不清楚 JS Macro 和 VBA 的区别。

**近期（2026年初）**：能独立设计参数三级分类体系、叠层墙拆解方案、清单匹配条件结构；能向 Revit API 文档索引具体的 C# 类和方法；能用 RevitLookup 验证 API 对象行为。

### 2. 文档能力：从执行者到设计者

**早期**：请 Claude 帮优化文字表达，聚焦在措辞和格式。

**近期**：设计文档结构本身，明确"产品经理看完整需求 / 开发看技术细节 / 测试看快速数据"的三版并行策略，然后交由 Claude 执行输出。

### 3. 提问方式的变化

**早期提问风格**：「这个参数是什么意思？」「这个功能怎么用？」

**近期提问风格**：「我打算用 X 方案解决 Y 问题，但我担心 Z 情况下会出问题，你觉得这个判断对吗？」→ 从知识获取型转向方案验证型。

### 4. 系统设计视角的建立

从只关注单个功能点，到能俯瞰整个「族库 → 品目 → 清单 → 工程量」的数据流动链路，并能识别链路上的断点和边界条件（如复合构件、衍生清单、几何参数保护）。
