### AI 工具在工作流中的角色

**不是替代编码，而是加速决策与实现的全链路覆盖：**

- **方案设计阶段**：用 Claude 对比多种技术路线（子域名 vs IP:Port、K-means vs 多层架构），并主动联网验证结论可靠性后再向上汇报

- **文档阶段**：AI 转录会议音频 + Claude 生成结构化会议纪要（已处理 10+ 场会议）
- **内容生成阶段**：设计 AI 图片/视频生成提示词策略，完成创意内容制作

### 自主设计的 Prompt 策略与工作流

- **YDC 语义匹配工作流**：将 LLM 定位在"语义理解层"而非直接替代整体算法，搭配编辑距离作为粗筛层，形成可解释、可验证的三层架构
- **AI 图像生成策略**：通过实验总结出 Seedream 模型特征——自然语言描述优于关键词堆砌、避免负面描述、不依赖精确数值约束；确立纪实风格数字插画为基调
- **AI 视频提示词设计**：确认简单镜头运动（静止镜头+帧内运动）优于复杂弧形/甩镜，规避 AI 视频工具技术瓶颈
- **会议纪要生成 SOP**：设计转录文本 → 说话人识别 → 错误纠正 → 结构化 .docx 输出的标准工作流，含行动项表格、待确认事项表格和专有名词修正说明

### AI 工具带来的效率提升

- YDC 非标清单分析：将手工逐条对比替换为 LLM 批量语义判断，**覆盖约 13,500 条记录**，估计节省业务人员数周工作量

## 三、遇到的技术难题与解决过程

### 难题 1：YAO 系统 API 逆向工程
**问题**：需要自动化从公司外包管理系统（YAO）批量拉取数据，但供应商无 API 文档。

**解决过程**：
- 通过浏览器 DevTools Network 面板逐步分析请求链路
- 识别出 Spring Boot + Apache Shiro 认证体系：需先 `GET /api/publicKey` 获取 RSA 公钥，使用 PKCS1_v1_5 加密凭证，`POST /api/login` 后从响应中提取 `authToken`
- 后续请求使用 `Authtoken: <token>` 头（而非标准 `Authorization: Bearer`），避开了常见误区
- 最终产出可直接使用的 Python 脚本（`requests + pycryptodome + pandas`）和 API 对接文档

**结论**：主动放弃等待文档的路径，选择自主逆向工程，节省沟通周期。

---

### 难题 2：LLM 在中文短文本相似度判断中的选型
**问题**：YDC 非标清单库中的条目均为 5–15 字的工程造价专业名词（如"消防工程费"vs"消防工程费用"），K-means 算法因无法处理领域语义失效。

**解决过程**：
- 研究 K-means 在短文本上的局限：无监督分簇无法确定最优 K 值，特征向量对专业术语歧义无感知
- 提出三层架构：编辑距离粗筛（字符级） → LLM 语义理解（含行业上下文） → 人工验证（领域专家兜底）
- 在监控工作流中将触发节点锁定为"定稿草案"，解决了因前置节点引发的重复入库 bug

---

### 难题 3：内部平台部署架构选择
**问题**：部门主管建议沿用供应商的 IP:Port 方案以保证运维一致性，但该方案存在安全性与可维护性问题。

**解决过程**：
- 联网验证技术主张：确认国内主流互联网公司（百度、京东、淘宝、腾讯）均以 Nginx 反向代理为标准
- 将论据定性为**行业惯例与企业级平台成熟度标准**，而非个人偏好
- 输出正式技术对比备忘录（含方案对比表、行业研究引用、numbered 建议），以"提供调研建议、决策权在领导"的姿态提交，成功推动讨论
- 关键说服框架：IP:Port 模式是外包交付标准，不是企业内部平台标准

---

### 难题 4：Wendy's 品牌色彩的验证
**问题**：需要为 Wendy's 来访展示设计易拉宝，不同来源的品牌色彩数据冲突（PMS 186C vs PMS 1795C），且印刷色与屏幕显示色存在显著差异。

**解决过程**：
- 多轮联网检索后确认：**没有任何第三方来源拥有经过验证的 Wendy's 官方 VI 指南访问权限**，所有数字均为从 logo 图像提取，不可作为印刷依据
- 识别到 Wendy's 红色（#E2203D）位于 RGB 色域溢出区间，CMYK 转换后会偏粉，需在 Illustrator/Photoshop 中以 CMYK 模式重建印刷文件
- 向上汇报时明确区分"设计预览稿"与"印刷终稿"两个阶段，避免直接将 HTML 原型发送印刷厂

---

### 难题 5：AI 视频生成中的技术瓶颈识别
**问题**：麦芽汇项目中需要使用 AI 生成情绪叙事视频，弧形镜头、甩镜等复杂镜头运动生成效果差，且 Ghibli 风格图片无法产出高质量视频帧。

**解决过程**：
- 系统研究 Seedream 5.0 Lite 模型机制，确认其 chain-of-thought 推理前置特性，调整 Prompt 策略
- 通过多轮实验识别出核心规律：**简单镜头运动携带更强情感力量**，改为静止镜头+帧内运动（灯笼摇摆、烟雾飘散、呼吸感）
- 风格从 Ghibli 动画改为纪实风格数字插画，保证视频生成输入帧的有效性
- 结合 首尾帧 模式（而非全能参考）完成三段视频制作

## 四、工作方式与思维方式

### 从业务出发理解需求

- YBP 项目中，主动深入学习麦当劳门店造价清单逻辑（族/组→品目→清单链路），在技术对接中成为业务与供应商之间的"翻译层"，而不是单纯的需求传递员
- 市场情报平台中，通过用户访谈（战略营销部）识别出"内联跟进备注"等体验痛点，主导产品功能迭代，而不只是实现最初的功能清单

### 技术决策时的权衡维度

- **验证优先**：推荐方案前先联网查证（如子域名方案、K-means 方案），避免用未经验证的技术主张向上汇报
- **安全与可维护性并重**：在 AI 代码审查中建立明确标准——能跑通 + 无死循环风险 + 无被攻击面（内外网区分处理）
- **逐步验证而非大跳步**：AI 图片/视频生成时坚持"生成一张验证方向，再批量生成"的小步迭代法

### 主动提出方案而非只执行

- 主动输出子域名反向代理方案备忘录，在无直接指令下提升了内部平台架构建议的专业度
- AI Coding 双周研究会的参与者角色 → 主动整理研究结论（米尺公司调研分析、Vibe Coding 工作流设计）
- 提出服务器采购 PPT 汇报逻辑框架（"部署什么 + 平台策略 + 推荐方案 + 费用对比"），而不是只列选项

### 沟通风格

- **自我批评而非甩锅**：YBP Q1 进度延误时，在绩效自评中主动标注"督促不力"为主因，而非将责任推给外部供应商
- **分层汇报意识**：CEO 级别的汇报 → 不提内部系统名称；上级 → 提供有据可查的建议；跨部门 → 用对方能理解的业务语言
