<task>
对项目进行全维度代码健康诊断（13 个维度），输出量化评分、完整问题清单和分批重构计划。
**只诊断不改代码**——改代码在后续用 `/task` 批量模式按计划执行。
</task>

<dimensions>

## 诊断维度（四层 13 维度）

### 结构层（影响面大，优先级高）

**D1 耦合度** — 改 A 会不会崩 B？
- 检查组件间的导入依赖数量和深度
- 查找跨组件直接引用内部状态
- 组件是否依赖过多外部模块

**D2 职责划分** — 每个单元是否只做一件事？
- 查找"万能文件"（> 300 行的组件）
- 组件内是否直接调 API、处理数据转换、包含业务逻辑

**D3 模块边界** — 模块之间是通过接口通信还是深入内部？
- 查找被多个文件导入的"上帝模块"
- 检查模块是否暴露了内部实现

**D4 依赖方向** — 依赖关系是否合理？
- 检查循环依赖
- 检查共享代码是否独立

### 实现层（局部优化，逐步改进）

**D5 代码重复** — 近似逻辑是否散落多处？
- grep 相似的函数签名和代码块
- 应该抽成公共组件/util 但没有的

**D6 错误处理** — 是否一致且完整？
- 检查 try-catch 使用
- 查找静默失败

**D7 类型安全** — 类型系统是否被正确使用？
- 纯 JavaScript 项目：检查 prop 验证、JSDoc 注解情况
- 潜在的 undefined/null 访问风险

**D8 性能隐患** — 是否有明显的性能反模式？
- 组件内创建对象/函数导致不必要 re-render
- 未清理的定时器/订阅

**D9 测试覆盖** — 关键操作链路是否有测试？
- 是否有自动化测试
- PDF 导出功能是否可测试

### 卫生层（认知负担）

**D10 死代码** — 是否有不再使用的代码？
- 未使用的函数、组件、导入、变量
- 注释掉的代码块

**D11 一致性** — 同一件事是否用同一种方式做？
- 命名风格是否统一
- 错误处理/日志格式是否统一

### 战略层（投入产出比）

**D12 代码热点** — 哪些代码改动最频繁且质量最差？
- 分析 git log 找改动频率最高的文件
- 交叉对比代码质量

**D13 知识孤岛** — 是否有只有一个人碰过的关键模块？
- 单人项目跳过此维度

</dimensions>

<workflow>

## Step 0: 读取项目上下文

```bash
echo "=== 代码健康诊断 $(date '+%Y-%m-%d %H:%M') ==="
```

读取（如存在）：
1. `CLAUDE.md`（技术栈、约束、完成标准）
2. `docs/architecture/`（设计意图基准）
3. 上一次诊断报告 `docs/reports/diagnose-*.md`

## Step 1: 探索项目结构 → 决定扫描策略

```bash
# 源文件统计
find src -type f \( -name "*.js" -o -name "*.jsx" \) | wc -l

# 顶层目录结构
ls -d src/*/
```

本项目 < 50 源文件，主 Agent 直接扫全维度。

## Step 2: 热点分析（可选）

```bash
# 最近 3 个月文件改动频率 Top 20
git log --since="3 months ago" --name-only --pretty=format: | sort | uniq -c | sort -rn | head -20

# 文件行数排序
find src -type f \( -name "*.js" -o -name "*.jsx" \) -exec wc -l {} \; | sort -rn | head -20
```

## Step 3: 全维度扫描

按 D1-D13（跳过 D13）逐维度扫描。对每个维度：
1. 用 Grep/Glob 做模式匹配
2. 读取关键文件做语义分析
3. 每个问题记录：`[维度] [文件:行号] [P0-P3] [置信度] 描述 | 依据`

**React 专项追加检查**：
- 组件 > 200 行
- `style={{}}` 内联样式
- props drilling > 3 层

## Step 4: 汇总评分

每维度评分 0-10（10 = 完美）。
综合健康度 = 13 维度加权平均（结构层权重 ×1.5）。

## Step 5: 生成重构计划

将 P0-P2 问题分批：
1. 同一模块的问题合并到同一 Batch
2. 有依赖关系的 Batch 标注前置条件
3. 每个 Batch 预估不超过 1 个会话
4. 每个 Batch 有明确验收标准

## Step 6: 输出报告

```bash
mkdir -p docs/reports
```

写入 `docs/reports/diagnose-YYYY-MM-DD.md`

## Step 7: 输出确认

```
✅ 代码健康诊断完成

综合健康度: X.X/10
扫描范围: [scope]
扫描文件: [N] 个
发现问题: P0 [N] 个 | P1 [N] 个 | P2 [N] 个 | P3 [N] 个
重构计划: [N] 个 Batch

报告: docs/reports/diagnose-YYYY-MM-DD.md

下一步：
- 查看报告，确认优先级排序
- 按 Batch 顺序用 /task 批量模式执行重构
- 全部完成后再次 /diagnose 验证改善
```

</workflow>
